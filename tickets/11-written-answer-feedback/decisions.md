# 11 — Written-Answer Feedback: Locked Decisions

Record of decisions resolved via `/grill-me` on 2026-07-17 (no open questions taken back to the
user; every branch resolved by the agent's own recommendation, per user instruction). Source of
truth for the written-answer-feedback ticket, alongside [plan.md](plan.md).

| # | Decision | Choice | Rationale |
|---|----------|--------|-----------|
| 1 | LLM call shape | Structured/tool JSON call, same injectable/mockable client pattern as ticket 06. Input `{question, gold_answer, user_answer}`; output schema `{outcome: enum[perfect, good, wrong], feedback: str}` enforced via the tool's JSON schema (`outcome` as a 3-value string enum) | Reuses the exact seam ticket 06 already establishes instead of inventing a second LLM-calling convention |
| 2 | Prompt/instructions shape | A short system/tool-use instruction: grade the `user_answer` against `gold_answer` for the given `question` using three bands only — `perfect` (fully correct, nothing material missing or wrong), `good` (substantially correct, minor omission/imprecision), `wrong` (materially incorrect, missing the key point, or off-topic/empty). `feedback` is 1-2 sentences, addressed to the learner, explaining the gap (or affirming correctness) — never a restatement of the gold answer alone | Keeps grading criteria explicit and stable so outcome mapping to SM-2 grades is meaningful; short feedback keeps the review UI compact |
| 3 | Malformed-`outcome` handling | If the tool call returns and parses as JSON but `outcome` is not exactly one of `perfect`/`good`/`wrong` (wrong type, extra value, missing field) — treated identically to a call failure per plan.md: fall back to manual flip-and-grade for that card. Same fallback for a raw call error (timeout, network, non-2xx, unparseable JSON) | Plan.md already specifies "malformed output → treat as a call failure"; this decision makes explicit that schema-valid-JSON-but-wrong-enum-value counts as malformed too, so there's one failure path, not two |
| 4 | Written-answer toggle scope | Per-review-session, not per-card and not a global setting. A toggle in the review UI (persists for the duration of the current review session only, defaulting to off/flip-and-grade at the start of each new session) | Plan.md says "per-review choice ... not a global setting"; per-session (not per-card) avoids forcing the user to re-toggle on every single card while still not persisting as a permanent account-wide setting, which ticket 10 (settings) doesn't own for this ticket |
| 5 | UI flow | (1) Card front shown with a free-text textarea + "Submit Answer" button instead of "Show Answer", when written-answer mode is on. (2) On submit, the textarea locks (read-only) and a loading state replaces the submit button. (3) On LLM response: reveal the gold answer, the outcome badge (perfect/good/wrong), and the `feedback` text, plus the four grade buttons (Again/Hard/Good/Easy) with the mapped button pre-selected/highlighted. (4) User clicks "Confirm" (or clicks a different grade button, which becomes the new selection) to apply the grade and advance to the next card | Matches plan.md's "shown alongside the revealed gold answer before the grade is (auto-)applied — user can still override" with a concrete, minimal number of clicks: one to submit the answer, one to confirm/override the grade |
| 6 | Override interaction | All four grade buttons are always shown (same four buttons as flip-and-grade, so there's one mental model for grading across both modes) with the mapped button visually pre-highlighted/selected as the default. Clicking a different button simply changes the selection — there is no separate "this seems wrong, override" secondary control | Reuses the existing four-button grading UI from ticket 09 verbatim rather than adding a new interaction pattern; "Hard" is reachable this way even though written-answer mode never auto-selects it, satisfying plan.md's "Hard stays reachable only via manual grading" |
| 7 | Grade application trigger | The grade is applied (via the ticket-08 persistence helper) only on the user's explicit confirm/click of a grade button, not automatically the instant the LLM responds — "auto-applied" in plan.md means "the mapped grade is pre-applied as the default selection," not "submitted without user action" | Keeps a human-in-the-loop confirmation step before any DB write, consistent with ticket 09's flip-and-grade flow where the grade is likewise only written once the user clicks a grade button |
| 8 | LLM call timeout/latency handling | A fixed client-side timeout (30s, matching the outer bound implied by ticket 06's background-job pattern) on the LLM call. While in flight: submit button shows a spinner/"Grading..." state, textarea is locked, no other action available on that card. On timeout: treated as a call failure (decision #3) — falls back to manual flip-and-grade for that card, with a brief inline notice ("Grading is taking too long — showing the answer instead") | The review flow is synchronous/interactive (not a background job like ticket 06's generation), so a bounded wait with a clear fallback keeps the UI from hanging indefinitely without needing polling/background-task infrastructure |
| 9 | Integration with ticket 09's grading endpoint | No new grading endpoint or persistence path. The written-answer flow's confirm action calls the exact same apply-grade call ticket 09's flip-and-grade buttons call (ticket-08's persistence helper), just triggered from a different UI control and pre-seeded with a suggested grade instead of requiring the user to pick one from a blank state | Plan.md: "Applies the resulting grade through the existing ticket-08 scheduler/persistence helper — no new grading path, just a new input source for the grade" |
| 10 | Outcome → grade mapping enforcement | The mapping (`perfect → Easy`, `good → Good`, `wrong → Again`) is applied purely client/view-side to decide which of the four existing grade buttons is pre-highlighted; the backend grading endpoint is unaware written-answer mode exists — it just receives whichever grade button was ultimately confirmed | Keeps the backend grading surface (from ticket 08/09) completely unchanged; written-answer feedback is purely an additive UI layer on top of it, per decision #9 |

## Notes
- This ticket adds no new DB columns/tables of its own beyond what 06 (LLM client boundary) and
  08/09 (scheduler, `reviews` audit row) already provide — the LLM outcome/feedback are ephemeral
  (shown in-session, not persisted) since only the resulting grade and `reviews` row matter for
  scheduling.
- The per-session toggle (#4) is pure client/view state (e.g. a query param or session-scoped flag
  on the review page), not a DB-backed setting — no migration needed for this ticket.
