# 14 — Sidebar Navigation & Note Content Pane: Locked Decisions

Record of decisions resolved directly by the agent's own reasoned recommendation (per user
instruction: "don't ask me any questions just go with your recommendation" — no interactive
`/grill-me` interview was run; every branch below was decided here instead). Source of truth for
this ticket.

| # | Decision | Choice | Rationale |
|---|----------|--------|-----------|
| 1 | Tree loading strategy | Subjects+folders shell stays eager-loaded (reuses ticket 04 issue #88's single-query eager-load pattern); the notes list under a given folder loads lazily via HTMX only when that folder is expanded | A user's subject/folder count is small and cheap to load up front (already proven at ticket 04); note counts per folder can grow large, so fetching every folder's every note on initial page load doesn't scale — lazy-load on expand avoids that |
| 2 | Note selection mechanism | Clicking a note fires an HTMX `hx-get` to a new note-content route, swapping only the right pane; the request also does `hx-push-url` so the browser URL becomes `/sources/{id}` | No-reload navigation consistent with the rest of the app's HTMX architecture; `hx-push-url` gets bookmarking/refresh/back-button support "for free" without hand-rolling client-side routing |
| 3 | Direct/deep link to a note | New `GET /sources/{source_id}` full-page route renders the whole shell (sidebar + this note pre-selected and expanded to reveal it in the tree) | Refreshing or sharing a note URL must work, not just the HTMX-swap path; requires a full-page variant of the same content |
| 4 | New note-content fragment route | New `GET /sources/{source_id}/content` returns just the right-pane fragment (used by the HTMX click-swap; the full-page route above reuses this fragment internally) | Standard "full page + matching fragment" pair, same pattern used throughout tickets 04-13 |
| 5 | New notes-per-folder fragment route | New `GET /folders/{folder_id}/notes` returns the lightweight sidebar list (note names + ids only, no content/actions) | Keeps the lazy-loaded sidebar fragment cheap; content and actions only load once a specific note is actually selected (decision #4) |
| 6 | "Convert all formats to Markdown for display" | No new storage-layer conversion. `sources.raw_text` already stores plain extracted text uniformly regardless of original format (PDF/TXT/MD, per ticket 05) — for `.md` sources that text already *is* Markdown source; for `.pdf`/`.txt` sources, blank-line-separated plain text is itself valid (if unstyled) Markdown source. Run `raw_text` through one Markdown-to-HTML renderer for all three formats at display time; no per-format branching needed | Simpler and strictly sufficient: the normalization the user asked for happens naturally at the rendering layer without a separate conversion step or new storage format, since Markdown is a superset of plain text for this purpose |
| 7 | Markdown rendering library | Add the `markdown` PyPI package (stdlib-adjacent, mature, one-call `markdown.markdown(text)` API) as a new dependency; render server-side in the note-content route/template | Keeps rendering server-side and Jinja-native, consistent with the rest of the app (no client-side JS Markdown library, no SPA framework) |
| 8 | Rendered-HTML safety | Use the library's default safe mode (no raw-HTML-passthrough extension enabled) | Notes are user-uploaded content rendered back to that same user; default escaping is the simple, sufficient choice for v1 — not treating this as a full security review, just a sane default |
| 9 | Relocating existing per-note actions | Convert-to-Flashcards status widget (`_source_status.html`), the Quiz Me button/flow, and a "View Cards" link (to ticket 07's existing `/sources/{id}/cards` page) all move into the right pane, scoped to the currently-selected note. Their underlying routes/logic are unchanged — only where they're rendered from moves | These are existing, working features (tickets 06/07/12); this ticket is a navigation/layout change, not a rebuild of their logic |
| 10 | Relocating structural CRUD | Create/rename/delete subject, create/rename/delete folder, and the upload-a-note form all move into the sidebar tree at their respective levels (e.g. a "+" control per level, inline rename/delete on each row) | These already exist (ticket 04, ticket 05); relocating them into the tree is layout-only, matching where a user now expects to find them in a sidebar-nav pattern |
| 11 | Styling approach | Extend ticket 13's existing `src/memory_ai/static/styles.css` (same custom properties: `--bg`/`--surface`/`--accent-primary`/`--accent-danger`/`--text`/`--border`) with a two-column flex/grid layout (fixed-width sidebar, flex-grow right pane); no new stylesheet, no CSS framework, no JS framework | Consistency with the just-established palette/approach; this app has no build step and no reason to introduce one for a layout change |
| 12 | "Use the Claude CSS plugin if needed" | No such tool exists in this agent's available toolset for a server-rendered Jinja/HTMX application (the closest match, the `artifact-design` skill, is scoped to the Artifacts feature — standalone HTML/Markdown documents rendered in a sandboxed viewer — not to this project's actual deployed UI). Proceeding with decision #11's plain CSS-custom-properties approach instead | Flagging this explicitly rather than silently ignoring the instruction: there isn't a matching tool to invoke here, so the closest sound equivalent (the project's own established styling system) is used |
| 13 | Responsive behavior | Desktop-first two-pane layout; below a small-viewport breakpoint (~768px) the sidebar collapses behind a toggle button rather than an always-visible column | Minimal, cheap responsiveness without deep investment; full mobile redesign is out of scope for this ticket |
| 14 | Empty states | Right pane shows a placeholder ("Select a note to view it") until a note is clicked; sidebar levels keep their existing empty-state messages (ticket 04's "no subjects yet" / "no folders yet" patterns), extended to a matching "no notes yet" at the note level | Consistency with the empty-state pattern already established at every other level of this hierarchy |
| 15 | Ownership scoping | Both new routes (`GET /sources/{id}`, `GET /sources/{id}/content`) and the new `GET /folders/{id}/notes` route resolve ownership via the same join-to-owner pattern as tickets 04/07 (404, never 403, for not-owned or nonexistent resources) | Matches the locked cross-ticket convention already established and tested throughout this codebase |

## Notes
- No database schema change — `sources.raw_text` and `sources.file_type` already exist and are
  sufficient (ticket 05).
- The existing `/subjects` route's template output changes (new sidebar shell replaces the old
  flat inline list), but the route itself, its auth requirements, and the underlying
  subjects+folders query are unchanged from ticket 04.
- `_folder_sources_section.html` (the old inline notes+quiz+convert list) is superseded by the new
  `GET /folders/{id}/notes` (sidebar) + `GET /sources/{id}/content` (right pane) pair and should be
  retired once the new routes cover its responsibilities, rather than kept alongside them.
